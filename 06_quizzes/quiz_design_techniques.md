### Q15

A CDC analysis tool (e.g., Questa CDC or Synopsys SpyGlass CDC) reports a "reconvergence" CDC
violation on a multi-bit bus crossing between two asynchronous domains, even though each bit
has its own 2FF synchroniser. Why is this flagged as a violation, and what is the correct fix?

A) The tool is reporting a false positive — independent 2FF synchronisers on each bit are
   the correct solution for multi-bit buses and no further action is required.  
B) Reconvergence occurs when multiple independently synchronised bits are combined in
   downstream logic before all bits have had time to resolve from metastability. The correct
   fix is to use a single handshake protocol (req/ack), a Gray-coded encoding, or an
   asynchronous FIFO so that all bits cross as a coherent group.  
C) The tool is flagging a clock frequency mismatch — the fix is to apply a multicycle path
   constraint of 2 on the crossing to give all synchronisers equal time.  
D) The fix is to add a third synchroniser flip-flop to each bit path to increase the mean
   time between failures (MTBF), which eliminates the reconvergence risk.

## Answer Key

### A1: B

A 2FF synchroniser is the standard circuit for single-bit CDC. The first flip-flop may go
metastable when sampling the asynchronous input, but is given one full clock period of the
destination domain (1/200 MHz = 5 ns) to resolve before its output is sampled by the second
flip-flop. With properly sized flip-flops, the probability of metastability propagating past
the second stage is negligible (MTBF >> system lifetime). This is fundamental FPGA CDC practice.

*Why A is wrong:* a single flip-flop provides no resolution time — if the first FF goes
metastable, its metastable output immediately drives downstream logic.  
*Why C is wrong:* a FIFO is appropriate for multi-bit data buses, not single control signals.
A FIFO with 4 entries addresses depth/rate matching, not the single-bit synchronisation problem.  
*Why D is wrong:* single-bit signals absolutely CAN glitch during transitions, and more
importantly, the issue is metastability — the flip-flop may not reliably capture '0' or '1'
regardless of glitches.

---

### A2: B

Metastability occurs when a flip-flop's setup or hold time is violated, placing the internal
latch node at an indeterminate voltage level (typically between V_IL and V_IH). The output may
linger in this state for a duration that is theoretically unbounded but statistically rare.
Eventually the flip-flop resolves to either '0' or '1', but different downstream paths may
sample the output during different phases of the resolution, potentially causing inconsistent
data capture. Metastability cannot be fully eliminated — only managed by providing sufficient
resolution time.

*Why A is wrong:* the output does eventually resolve to a valid value (either '0' or '1'); the
issue is the indeterminate intermediate state and the resolution time, not a permanent error.  
*Why C is wrong:* missing a clock edge due to jitter is clock capture failure, not metastability.  
*Why D is wrong:* conflicting outputs from two flip-flops in the same CLB is a multi-driver
contention issue, not metastability.

---

### A3: C

Xilinx recommends synchronous resets for most UltraScale designs. UltraScale flip-flops have
a dedicated synchronous set/reset (SR) port that maps efficiently to the CLB structure.
Asynchronous resets, by contrast, route on a separate asynchronous reset network, consume
additional routing tracks, and can cause timing issues because the reset-release path must
also meet timing (synchronous release with an asynchronous reset synchroniser). Applying reset
only to registers that need it avoids consuming the SR port on flip-flops whose state is
irrelevant at startup (e.g., pipeline registers in the middle of a datapath).

*Why A is wrong:* asynchronous active-low resets require extra logic inversion, route on
asynchronous reset nets, and create reset-removal timing problems.  
*Why B is wrong:* while active-high is slightly more efficient in UltraScale (the SR port is
active-high), asynchronous style still has the routing and timing disadvantages described above.  
*Why D is wrong:* relying solely on INIT attributes is dangerous because INIT applies only at
power-on/configuration. If the design uses GSR (Global Set/Reset) or if a functional reset is
needed at runtime (e.g., after a soft reboot), there must be a real reset network.

---

### A4: B

Pipelining breaks a long combinational path into shorter segments by inserting registers at
intermediate points. Each stage can now complete in a shorter time (one register-to-register
path), so the maximum clock frequency increases. However, a piece of data now takes more clock
cycles to pass through all stages. For a 4-stage pipeline, the first output appears after 4
clock cycles (latency = 4 cycles). Once filled, the pipeline produces one result per clock
cycle (full throughput) — a classic throughput/latency trade-off.

*Why A is wrong:* pipelining INCREASES latency — more cycles are needed before the first
result appears.  
*Why C is wrong:* adding pipeline registers increases flip-flop count (resource usage increases,
not decreases). Latency also increases, not decreases.  
*Why D is wrong:* pipelining is essential at 500 MHz, 1 GHz, and any frequency where the
combinational path is the bottleneck. Frequency is not a threshold below which pipelining
has no effect.

---

### A5: B

Clock outputs from IP cores (such as MMCM/PLL output clocks in a Clocking Wizard) must be
connected to a global clock buffer (BUFG or BUFGCE) before being used to clock logic. This
ensures the clock is distributed on the low-skew global clock network. Routing a clock through
fabric interconnect introduces high and unpredictable skew, which degrades hold margins and
can cause functional failures. Vivado's DRC (CLOCK_007) will warn if a clock source drives
logic without a clock buffer.

*Why A is wrong:* routing a clock through fabric interconnect introduces skew comparable to
data paths — this is a known DRC violation in Vivado.  
*Why C is wrong:* using a GPIO pin and external buffer is unnecessary and wastes board resources;
the FPGA has dedicated internal clock buffers for exactly this purpose.  
*Why D is wrong:* IP clock output ports are specifically designed to drive downstream clock
networks — they are valid clock sources when properly buffered.

---

### A6: B

In a binary counter, multiple bits change simultaneously at rollover transitions (e.g., 0111
→ 1000 changes all four bits). When passing a binary counter through a 2FF synchroniser
(each bit independently synchronised), different bits may be captured from different phases of
the multi-bit transition, producing invalid intermediate codes (e.g., 1000 + 0111 → 1110 = 14,
which was never a valid count). A Gray code ensures exactly one bit changes per transition,
so the synchronised value is always either the old or new valid count — never an intermediate
invalid code.

*Why A is wrong:* Gray-code counter frequency is limited by the XOR decode logic required for
arithmetic — if anything, it is slightly slower than a binary counter at high frequencies.  
*Why C is wrong:* the 2FF synchroniser captures the raw Gray-code bits. The decoder must be
in the destination domain and handles the Gray-to-binary conversion after synchronisation.  
*Why D is wrong:* a Gray-code counter requires the same number of flip-flops as a binary
counter — one per bit.

---

### A7: B

A reset synchroniser allows an asynchronous reset to be asserted immediately (asynchronously,
for safety at power-on) but released synchronously (to avoid hold violations on reset removal).
The circuit: D inputs of both FFs tied to '1', asynchronous reset applied to both FF reset
ports. When reset is asserted, both outputs immediately go to '0'. When reset releases,
the '1' propagates through the two FFs over two clock cycles, ensuring a clean synchronous
reset-release edge.

*Why A is wrong:* direct asynchronous reset distribution causes reset-removal (deassertion)
to be asynchronous with the clock. If reset releases at a bad time relative to the clock edge,
flip-flops can exit reset in metastable states, especially in high-speed designs.  
*Why C is wrong:* debouncing applies to mechanical buttons/switches, not to logic signals.
A power-good signal from a PMIC is a clean electrical signal; the issue is synchronisation,
not debouncing.  
*Why D is wrong:* STARTUPE2 is a configuration/startup primitive used to access the
configuration interface, not for general-purpose reset synchronisation.

---

### A8: B

Resource sharing works by time-multiplexing: a single DSP or ALU is used to serve multiple
computation requests that cannot occur simultaneously. A multiplexer on the inputs selects
which operands are presented to the shared operator each clock cycle. An output register holds
each path's result until it is consumed. This requires that the operations for different paths
are scheduled to non-overlapping clock cycles — which is manageable if each path has a
throughput requirement less than or equal to the shared operator's clock frequency.

*Why A is wrong:* retiming relocates registers to shorten critical paths; it does not share
operators between data paths.  
*Why C is wrong:* replacing a multiplier with a LUT table trades one resource (DSP) for another
(LUT/BRAM) — it does not achieve sharing.  
*Why D is wrong:* `set_multicycle_path` relaxes timing constraints but does not change the
hardware structure. The operator is still replicated for each path in RTL — the synthesiser
does not automatically share operators based on timing constraints.

---

### A9: C

AXI4 interconnects have a configurable limit on the number of outstanding transactions
(in-flight read or write requests without responses). If the DMA issues more transactions than
the interconnect's outstanding transaction limit, backpressure is signalled via the AWREADY
or ARREADY signals. If the DMA does not handle this backpressure correctly (i.e., it continues
to issue transactions without checking READY), arbitration errors and FIFO overflow can occur
in the interconnect.

*Why A is wrong:* AXI ID width mismatch causes ID remapping issues (the interconnect widens
or narrows IDs), which can affect performance and ordering but does not directly cause
"arbitration errors."  
*Why B is wrong:* a clock domain mismatch between DMA and DDR controller would typically
be caught at IP Integrator validation (DRC) and would cause functional failure rather than
specifically "arbitration errors."  
*Why D is wrong:* AXI4 masters are explicitly designed to issue read and write transactions
to memory — the entire purpose of an AXI DMA is to read from and write to memory.

---

### A10: B

A bubble is a cycle in which a pipeline stage has no valid work to perform. It is represented
by deasserting the `valid` signal on the pipeline's dataflow interface. Bubbles reduce effective
throughput below the full pipeline rate and occur when: the source cannot produce data fast
enough (input starvation), a downstream stage exerts backpressure (the pipeline stalls), or
a conditional branch kills an in-flight instruction. Managing bubbles efficiently is a key
concern in high-throughput pipeline design.

*Why A is wrong:* a combinational loop is a different pathological design issue; the tool
inserts a buffer to break timing loops, not to introduce pipeline bubbles.  
*Why C is wrong:* constant propagation optimises out logic that always evaluates to the same
value — this is a synthesis optimisation, not a run-time pipeline concept.  
*Why D is wrong:* BRAM read latency is a fixed structural latency (typically 1–2 cycles),
which is accounted for in the pipeline design. It is not called a "bubble" — a bubble is a
valid/invalid flow-control concept, not a structural latency.

---

### A11: C

The key concern with a slow-domain pulse crossing to a fast domain via a 2FF synchroniser is
that the fast domain samples the signal multiple times during the single long pulse. If the
signal represents a one-shot event (e.g., "increment a counter once"), the fast domain will
see it asserted for approximately 100 ns / 4 ns = 25 cycles of the 250 MHz clock, causing
the counter to increment 25 times instead of once. The solution is to use a pulse synchroniser
(toggle synchronisation): convert the pulse to a level toggle in the source domain, synchronise
the toggle using a 2FF synchroniser, then detect the edge in the destination domain to
re-create a single-cycle pulse.

*Why A is wrong:* while 100 ns > 2 cycles of 250 MHz (8 ns) is correctly identified as
sufficient for the synchroniser to capture the value, the answer misses the critical issue of
multi-sampling the pulse.  
*Why B is wrong:* the 2FF synchroniser is NOT always safe — it must be paired with appropriate
pulse-width management. A pulse of exactly one fast-domain cycle would be missed by a 2FF
synchroniser with very high probability.  
*Why D is wrong:* a 2FF synchroniser works regardless of which domain is faster — the
requirement is that the source signal is stable for long enough to be captured, which depends
on pulse width, not relative frequency.

---

### A12: B

With a 200 MHz FPGA clock and a 10 MSps input sample rate, there are 200 MHz / 10 MSps = 20
clock cycles available to process each input sample. A single DSP48E2 slice can be time-multiplexed
to compute all 32 MAC operations for a 32-tap FIR within these 20 cycles — but 32 > 20, so
a simple single-DSP approach is not sufficient. In practice, designers use a small number of
DSPs in a folded architecture (e.g., 2–4 DSPs computing 8–16 taps each within the 20-cycle
window). This is called "systolic folding" or "time-multiplexed filtering" and is a standard
technique for trading area against throughput when the input data rate is much lower than the
clock frequency.

*Why A is wrong:* retiming improves timing closure but does not reduce the number of DSP
instances required to implement 32 independent multiplications.  
*Why C is wrong:* carry-save arithmetic reduces adder tree depth in software or ASIC contexts
but does not apply to DSP48 slice sharing in FPGA design.  
*Why D is wrong:* coefficient quantisation reduces the precision of the filter, which
degrades filter performance. It may allow coefficients to share DSP operand slots in very
specific cases, but it is not a resource-sharing technique in the general sense and
degrades the design.

### A13: C

The most insidious risk of mixing resets is non-uniform reset release timing. Asynchronous
resets de-assert based on the reset signal's timing relative to the clock edge. Different
flip-flops with the same asynchronous reset may release on different clock edges if the reset
deassertion occurs near a clock edge and flip-flops are in different physical locations (slight
clock skew differences). This causes some registers to be "live" one or two cycles before
others, and if downstream combinational logic samples a partially-reset bus, it may capture
corrupt values. This is prevented by the reset synchroniser pattern (see Q7).

*Why A is wrong:* synthesis tools support both synchronous and asynchronous reset flip-flop
primitives and can handle mixed designs without failing.  
*Why B is wrong:* synchronous resets complete within one cycle by definition (the reset value
is clocked in on the next edge). The scenario described applies to asynchronous reset release,
not synchronous.  
*Why D is wrong:* mixing reset styles is NOT standard practice at scale — it is a known
source of subtle bugs and is actively discouraged by Xilinx and most FPGA design guidelines.

### A14: B

When a customised IP's data width parameter is changed in IP Integrator, the tool should
auto-propagate the new port width to connected interfaces if the block diagram is re-validated.
The most common cause of the inconsistency is that the block diagram was not re-validated
(`Run Connection Automation` or `Validate Design`) after the parameter change, leaving the
AXI interconnect's internal parameter referencing the old value. Vivado IP Integrator uses a
live parameter resolution system — stale diagrams can show mismatched widths until revalidated.

*Why A is wrong:* deleting and re-adding the interconnect would work, but it is not the most
likely cause and is a destructive approach. Re-validation is always tried first.  
*Why C is wrong:* Vivado IP Integrator fully supports AXI data width changes on standard
Xilinx IP — this is one of its primary features.  
*Why D is wrong:* `C_DATA_WIDTH` on standard Xilinx AXI IPs (e.g., AXI BRAM Controller,
AXI DMA) controls the width of the AXI data bus, not just an internal FIFO.

---

### A15: B

Reconvergence CDC is a structural hazard where multiple bits, each individually synchronised
by their own 2FF synchroniser, are consumed together in combinational logic in the destination
domain. The individual synchronisers resolve their metastability independently — bit 0 may
resolve 1 ns after the clock edge, bit 7 may resolve 3 ns after the clock edge. If the
destination logic samples all bits together (e.g., a comparator or decoder), it may observe
bit 0 in its new state while bits 1–7 are still in the old state — an impossible intermediate
bus value. The correct fix is to guarantee bit coherence by using a handshake (where the
sending domain signals "all bits valid" and the receiving domain samples all bits together),
a Gray-coded encoding (for counters), or an asynchronous FIFO (for arbitrary data).

*Why A is wrong:* independent 2FF synchronisers on each bit are explicitly NOT the correct
solution for multi-bit buses. This is the exact scenario the CDC tool is flagging.  
*Why C is wrong:* reconvergence is a structural coherence problem, not a frequency problem.
Multicycle path constraints do not fix structural CDC issues and should not be applied to
asynchronous crossings.  
*Why D is wrong:* adding a third synchroniser stage increases MTBF (reduces the probability
of metastability propagating) but does NOT solve the reconvergence problem. Even with three
stages, bits resolve at different times — the combinational-logic sampling hazard remains.
